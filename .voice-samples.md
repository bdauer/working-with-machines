## To a CS student asking what to learn

For many software development jobs knowing algorithms in the academic sense doesn't really matter. Learning CS is not learning software development. It hopefully teaches habits for thinking about problems and builds a strong knowledge base of fundamentals about how software works, classes of problems that will come up, how to analyze the effectiveness of a solution.

The fundamentals haven't really changed with AI. Break down problems (into actionable steps) and solutions (code should ideally be modular, not spaghetti), think about edge cases, understand the domain and the needs you're trying to meet, think outside of the box, learn to collaborate with SME's and stakeholders, and figure out the appropriate fit for the problem while balancing competing concerns. Soft skills matter.

With agentic workflows some of the process of research, planning, execution, can be thought about in the same way as code and architecture, via a higher order analysis. So it's still the same types of problems tackled, just at a different layer of abstraction. The more you work on different types of problems the more you begin to see similar shapes between them.

Since agentic output is inconsistent, and can be produced quickly, it continues to be important that one develops a good eye and understanding for what's actually correct. That takes practice, experience.

The answer used to be: stop worrying about foundations and build something. I don't know if that's enough anymore because building alone is easier than it used to be, so the bar gets raised. Understand what you're building. Don't stop building. Keep improving.

## Unfiltered thinking about paradigm shift

As we become more agentic my theory is, for those with high enough seniority to be trusted, we become more like principals or staff eng consulting on multiple teams working on different projects simultaneously. That's in direct conflict with the business-as-usual pre-agentic gates. It means we need to shift the gates where person-to-person consultation is needed up some levels to avoid significant slowdown. This is one perspective, I'm convinced we're in a radical paradigm shift and our profession is changing. We are not augmenting existing processes. They are becoming obsolete, but their shape hints at the outline of the future reality, with higher order analysis to recognize those procedural steps that can be codified into agentic workflows. The outcome once we've all made the leap looks very different than what we do today. This is not the only perspective, but if it is the correct one, what human touchpoints remain to ensure we have the right input there? Does a higher order review process meet our certification needs, instead of line-by-line, which is the language of the old world? How do we establish trust in outcomes on a per-peer basis to know what level of being hands-off is correct? Many unanswered questions that need input.

## Narrating an agentic process

I recognized that there are standards and best practices out there and the problem domain is common and well-researched. So I researched with Claude, having it spin up research agents on best practices and risks, cycles of gap analysis using the collaborative friction approach and more research (claude really is the best rubber duck ever). I looked for holes and I had claude do the same. I already had figured out a key pattern and didn't want to lose that, so pushing back on that as a gap and saying, hey claude let's figure out how to keep what works, meant it could synthesize a mechanical solution since it already had the pattern.

Once I was satisfied with the big picture I had it come up with a plan. This meant a bit more friction-based refinement. When I was satisfied, which included tweaks to the implementation and suggestions for multi-reviewer review cycles and fixes, it was off to the races. I had to insert myself a couple times. I'm never satisfied on a large chunk of work like this with the planned review and usually not with the test coverage, but I also know I'll get a summary after, and that summary is another friction point before I need to look at the code.

For something this big the major review dialogues (because there are often multiple) always means a few things: run simplify which has gaps because it looks at small details, run a larger architectural review for improvement opportunities framed as "leave the campsite better than you found it", and then some more based on context e.g. a technical writer to make sure docs are well updated, different angles of security review, etc. But there could also be friction I identify from the summary, and most certainly from the findings of the reviews. Once the high level output I'm seeing is satisfactory I'll start reviewing the full code output and if I see issues, I'll call them out as a type of gap it missed so that it can suggest a category of review that's needed for us to refine, not to fix a specific issue. That occurred here.

So basically with minimal insertion of myself I was able to let it run for 45-60 minutes or so on its own. When I prodded because I found an issue that could send it going for another 10-15 as it tried to resolve an issue, tested, found an appropriate workaround. In this case that also meant more cycles of simplify, architectural review, etc.

## On encoding process early — conviction, direct, no hedge

It's not premature to encode process early. Early is good because that gives friction. As soon as there's enough of something to work against and iterate on you're in good shape.

## On personalized skills as source material — hunch, directional

This will emerge from practice but my hunch is that a personalized skill as the source carries more meaningful signal than a generalized skill because it carries more friction.

## On the agentic developer and adoption — conviction, paradigm shift framing

It is a major challenge. The agentic developer moves faster with less concern for every individual line of code. They generate more documentation, can own multiple workstreams with appropriate peer-consultation. It's the avant garde and old guard expected in a period of paradigm shift.

## On higher-order review and compliance — certain, patient

I'm certain this is the direction we're heading but it's going to take time for regulatory bodies and cultural consensus to land there. See however the PR review skill as directionally aligned. PR review has human in the loop. Knowledge transfer is preserved. Some of the code is read. But much of the output is at a higher level with synthesis that would not be possible in a timely manner without agents.
